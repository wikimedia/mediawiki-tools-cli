# mwcli Dev Recipe YAML (draft)

This directory contains a draft recipe schema and example recipes derived from the copied MediaWiki-Docker extension setup pages.

Use with:

- `mw dev recipe --file /path/to/recipe.yaml`
- `mw dev recipe validate --file /path/to/recipe.yaml`

## Quick Start (CI / anonymous development)

Copy the included default config to your mwcli config directory before running
recipes. This sets HTTP-based Gerrit interaction (no SSH key needed) and shallow
clones (faster checkouts):

```sh
cp .recipe/mwcli.config.json ~/.config/mwcli/config.json
```

The config file contains:

```json
{
  "gerrit": {
    "interaction_type": "http"
  },
  "mw_dev": {
    "docker": {
      "shallow_clones": true
    }
  }
}
```

These values can also be set per-recipe in the `source:` block, but the config
file approach keeps recipes environment-agnostic.

## Schema (v0.1)

```yaml
type: mwcli.dev/recipe            # required, also accepts mwcli/recipe
version: 0.1                      # required
name: my-recipe
description: Optional human-readable text

source:
  useGithub: false                # optional
  shallow: true                   # optional, prefer mwcli config instead
  gerritInteractionType: ssh      # optional, defaults from mw config
  gerritUsername: addshore        # optional, defaults from mw config

env:
  SOME_ENV: value                 # optional, written to mwdd .env

customCompose:                    # optional, writes ${mwddDir}/<name>.yml
  name: custom
  content: |
    services:
      some-service: {...}

services:
  - name: mediawiki
    state: started                # started (default) or stopped
  - name: mysql

code:
  core: true
  extensions:
    - name: ContentTranslation
    - name: VisualEditor
  skins:
    - name: Vector

sites:
  - dbname: en
    dbtype: mysql                 # mysql, postgres, sqlite

jobRunner:                        # optional, manages mediawiki/jobrunner-sites
  sites:
    - en

localSettings:
  # Preferred: per-recipe files in LocalSettings.d/
  files:
    shared:                       # loaded for all wikis
      - content: |
          wfLoadExtension( 'Foo' );
    perWiki:                      # loaded only for matching $wgDBname
      en:
        - content: |
            $wgLanguageCode = 'en';

  # Legacy: append directly to LocalSettings.php (prefer files above)
  appendPHP: |
    wfLoadExtension( 'Foo' );

  yamlSettingsFile: settings/dev.yaml
  yamlSettings: |
    extensions:
      - Foo

maintenance:
  - name: update
    command: ["php", "maintenance/run.php", "update"]
    user: nobody                  # optional
    workingDir: /var/www/html/w   # optional
    env:                          # optional
      PHPUNIT_WIKI: en

# Stretch goal support (implemented as generic git fetch + cherry-pick)
patches:
  - name: some-gerrit-change
    repoPath: extensions/Foo      # path relative to MEDIAWIKI_VOLUMES_CODE or absolute
    fetch: ["https://gerrit.wikimedia.org/r/mediawiki/extensions/Foo", "refs/changes/34/1234/5"]
    cherryPick: FETCH_HEAD
```

## LocalSettings.d

Recipes can place per-recipe PHP files in `LocalSettings.d/` instead of
appending to `LocalSettings.php`. This is the preferred approach:

- **Shared files** (`localSettings.files.shared`) are written to
  `LocalSettings.d/recipe-{name}.php` and loaded for all wikis.
- **Per-wiki files** (`localSettings.files.perWiki`) are written to
  `LocalSettings.d/{dbname}/recipe-{name}.php` and loaded only when
  `$wgDBname` matches.

Files include a `// Generated by mwcli recipe` marker so `mw dev destroy`
can safely remove only recipe-managed files without touching user-created
configuration.

## Notes

- Recipes are intentionally **environment-agnostic** YAML and can be stored in extension repos.
- `localSettings.yamlSettingsFile` + `localSettings.yamlSettings` are included to align with MediaWiki's YAML settings format recommendation.
- Extension `composer.json` merge includes are managed via `composer.local.json` and applied with a single `composer update --with-all-dependencies` run.
- `jobRunner.sites` is preferred over explicit `runJobs.php` recipe steps.
- `patches` is intentionally generic so it can later map to higher-level Patch Demo/Gerrit/GitHub PR abstractions.
- The recipe engine automatically waits for sites to respond before running content creation steps.
- When `gerritInteractionType` is `ssh` but no username is configured, the engine falls back to `http`.
